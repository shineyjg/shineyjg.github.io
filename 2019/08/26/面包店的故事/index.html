面包店的故事

2019-08-26
  • 软件随笔
  • 面包店
  • 性能

从前有一家个性化面包店。它的生意模式是这样的：

  1. 来店前，顾客必须打电话预约，否则不会受到接待。

  2. 刚开始营业时，只有店主一人提供做面包服务。

  3. 已预约顾客到了面包店，需要在大堂有序等待被接待，一直等到他前面的所有顾客拿到他们各自定制的面包离开店里，才会轮到该顾客被服务。

  4. 被服务时，顾客提出要求，店主按顾客要求从库房取不同材料生产面包，取材料的平均时间是1分钟，烤面包的平均时间是1分钟，在等待面包完成的时候，顾客等着，店主也等着，但偶尔会接预约电话。

让我们首先看看在这个场景下的系统处理能力：

  • 电话只有一台，在预约通话中，别人无法打进电话，也就无法完成预约。同时接收预约的能力为1。

  • 大堂容量有限，只能容许10人等待。如果已经有10人在店中等待，第11人到来时，店主就会说：抱歉，暂时无法为你服务，请稍后再来，新来的顾客就走了。因此同时能接待最多顾客数：10。

  • 因生产面包的时间是2分钟，如果店里平均等待人数有5人，则每人从进店到离店的平均时间是10分钟，因为顾客需要等待前面4人得到服务后才能被真正服务。

如果业务壮大了，有哪些可能业务瓶颈？又有哪些优化措施呢？

  1. 店主生产面包的过程等着是浪费，他可以在等待前一个顾客烤面包的同时，为后面的顾客准备材料，这样并行操作，可以把效率提高1倍，在相同的时间内给更多的人提供服务。
  2. 店主一个人毕竟忙，他可以再雇一个员工，这样效率可以再翻倍。
  3. 一个电话在通话中，别人就无法打进电话预约，可以增加一个电话机，这样允许2个人同时预约。
  4. 大堂容量可以扩大一倍，这样可以容纳更多的人等待，同时接待顾客的人数也会翻倍。
  5. 改进取材料流程，减少取材料时间，可以显著提升平均响应时间
  6. 缩短烤面包时间，可以显著提升平均响应时间

研发的伙伴们，接下来请大家来思考下，是否能够对应面包店实体和相关技术术语：

  • 面包店实体： 老板 员工 顾客 打电话预约 电话机 大堂 库房 材料 烘烤 等待 ...

  • 技术术语：tcp连接 连接数 cpu 线程 请求 内存 数据库 请求队列 客户端 QPS 响应时间 数据组装 序列化 吞吐率 ...

能否改进这个故事？从而引入更多实体，最终看看对整个生意的影响（比如装材料的盆数量有限，如果盆都被占住了，就没法做面包了，这时候只能暂停所有工作，先把有些盆腾出来，洗干净，再继续之前工作）。

特别地，可以关注所做改进是影响了平均响应时间，还是只是提升了同时能处理的顾客数，甚至使平均响应时间增大了。
