Vue前端首页加载速度优化

2019-03-19
  • 前端
  • Vue
  • 首页加载
  • 优化

# 问题

某Vue前端应用首页加载速度慢：

上图是在模拟4G网络（4M带宽，20ms延迟）的情况下，chrome访问首页的截图，首次看到页面时间在7-8秒，总共加载时间22秒，总共流量5.4M。

初步结论：资源5.4M偏大，仔细看最大的一个请求是3.7M，仔细分析该文件内容，为webpack打包生成的js，里面包含了大量图片，也包含了非首页的js、图片资源（vue单页面应用特点）。

注：

这里为了精准测试，避免环境因素干扰。用到了chrome的incognito模式，并模仿4G网络减速访问，并禁用cache功能，以便在本地尽量公平测试性能并进行对比。

# 分析

vue-cli默认生成的项目已经包含webpack插件：webpack-bundle-analyzer，如果没有该插件可以自己添加：

var BundleAnalyzerPlugin = require('webpack-bundle-analyzer').BundleAnalyzerPlugin
webpackConfig.plugins.push(new BundleAnalyzerPlugin())

webpack 带--report参数构建时将生成buddle报告：

npm run build-test --report

这里是package.json对应的脚本：

  "scripts": {
    "dev": "cross-env URLCONFIG=pre webpack-dev-server --inline --progress --config build/webpack.dev.conf.js",
    "start": "npm run dev",
    "build": "npm run build-prod",
    "build-test": "cross-env URLCONFIG=test node build/build.js",
    "build-dev": "cross-env URLCONFIG=dev node build/build.js",
    "build-pre": "cross-env URLCONFIG=pre node build/build.js",
    "build-prod": "cross-env URLCONFIG=prod node build/build.js"
  }

构建后命令行输出，可以看到构建生成的大文件：

构建后还会自动打开浏览器页面，可以看到比较大的有两块，我们主要针对这两快进行优化：

从上图中可以看到可以优化的点：

  1. assets中的img太大
  2. views中包含了所有界面js资源
  3. 包含了element-ui和echarts依赖资源，从这个图中没有看到其它外部依赖是因为之前项目已经简单优化过，把vue、vuex、vue-router、mint-ui、axios、fecha等都已经通过外部引入的方式进行了优化，否则这个js还要大不少，但之前优化漏了element-ui和echarts。而且经过分析echarts原来是被项目中的v-charts依赖。

# 优化

# 1. 把assets中图片资源转成static中资源

assets中资源会被webpack打包到最终的js中，其效果是增加了js文件体积，而且使页面渲染时机推迟。因为页面需要在这个js下载完毕才会开始渲染，js文件大意味着下载时间长。图片转成static资源后，webpack不会打包static中图片到js文件中，js体积减少，下载时间加快。而且浏览器会并发加载js和这些图片，也加快了整体资源下载速度，从而提前了页面开始渲染的时间。

具体方法：

把类似代码：

<img style="cursor: pointer;" v-show="!indexHeaderChangeTheme" src="../assets/img/index/rubikt-logo-white.jpg"
          alt="logo" class="marginleft40 index-logo"/>

改成：

<img v-show="indexHeaderChangeTheme" src="/static/img/index/rubikt-logo-dark.jpg" alt="logo" class="marginleft40 index-logo" />

注：

webpack会把相对目录的图片进行打包转换，因此这里需要改成绝对路径，并把相应图片从assets目录移到static目录 有些小图片可以保留在assets中

# 2. 为了加快首页加载速度，启用webpack懒加载和分模块打包功能，把首页资源单独打包。这里更改routes.js:

const routes = [{
    path: '/',
    name: 'index',
    meta: {
      title: '互联网医院开放平台'
    },
    // component: lazyload('index'),
    component: r=>require.ensure([],()=>r(require('@/views/index')),'index'),
  },

改动在component这一行，主要是给一个分模块打包时的group名字，这里是index，只是把首页单独打包。

从上图可以看到：构建后输出已经有两个js，其中1.xxxx.js文件为首页对应的js，只有70k。

# 3. 把element-ui转成外部依赖：

在webpack.base.conf.js中配置externals：

externals: {
    'vue': 'Vue',
    'vue-router': 'VueRouter',
    'element-ui':'ElementUI',
    'mint-ui': 'mintUi',
    'vuex':'Vuex',
    'axios':'axios',
    'fecha':'fecha'
    }

在main.js中，需要注释掉原先对element-ui的相关依赖：

// import ElementUI  from 'element-ui'
// import 'element-ui/lib/theme-chalk/index.css';
// Vue.use(ElementUI)

在index.html中增加element-ui和v-charts的外部引用：

<link rel="stylesheet" href="https://unpkg.com/element-ui/lib/theme-chalk/index.css">
<script src="https://unpkg.com/element-ui/lib/index.js"></script>
<script src="https://cdn.jsdelivr.net/npm/echarts/dist/echarts.min.js"></script>
<script src="https://cdn.jsdelivr.net/npm/v-charts/lib/index.min.js"></script>
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/v-charts/lib/style.min.css">

经过上述调整后，再加--report运行构建：

从图中可以看到：

  • 左边js减小了体积，增加了右边方框中的1.xxxx.js对应的是首页的资源。
  • 0.xxxx.js中还是有很多大图片是因为到目前为止我只优化了首页的图片资源。
  • 1.xxxx.js中已经去除图片资源
  • element-ui和echart资源已经被排除在外

# 效果

可以看到优化后效果：流量为1.7M，加载时间总共7秒，首次看到渲染出的页面时间在3-4秒。

# Todo：其它优化

  1. 有些图片确实太大了，需要预先做压缩处理，再放到项目中。

  2. 除了首页，也可以优化其它页面

  3. 其它页面的大图片资源从assets转到static，可以进一步减少js资源大小，提高非首页加载速度

  4. extract代替external，把全依赖第三方库改成按需引用库

  5. 启用http2.0，利用http2.0多通道能力提升并发请求数，从而加快前端资源的加载

  6. 首页在上述模拟4M网络带宽的条件下，仍然存在渲染速度慢的问题。因为Vue是通过js在浏览器端渲染得到页面元素，可以通过服务端渲染SSR技术或客户端渲染CSR技术来进一步优化页面渲染速度。

我将在后续文档中分享SSR和CSR相关内容。
