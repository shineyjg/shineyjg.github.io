# 一、背景

  • 微服务系统中服务众多。
  • Hystrix+Turbine从横向角度，监控接口的可用性和性能。
  • Sleuth+Zipkin方案从纵向角度，监控调用链可用性和性能。

# 二、原理

# 1. sleuth

在请求和响应头添加header，使trace id和span id在请求和响应中传递，从而获取整个调用链成功失败信息及运行时间信息。

树形视角

注：发送和接收都分配了一个span id是因为：

  • 发送了不等于接收到（发送位于consumer端，接收位于provider端，中间是网络隔离的）
  • 接收到不等于开始处理（接收和处理线程往往是两个线程，如果处理线程忙，接收到的请求往往在队列中等待）
  • 这样可以精确统计：什么时候发送，什么时候接收，什么时候处理

# 2. zipkin

zipkin负责收集和显示、查找sleuth数据（也可以存储）

# 四、zipkin实战

# 1. 方式一：docker容器

在命令行中执行：

docker run --name zipkin -d -p 9411:9411 openzipkin/zipkin

注：

这里采用zipkin官方镜像运行zipkin容器，端口为9411，首次运行将从镜像仓库下载zipkin镜像

# （3）注解方式启用zipkin server

src/main/java/com/example/zipkin/ZipkinApplication.java:

package com.example.zipkin;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import zipkin2.server.internal.EnableZipkinServer;

@SpringBootApplication
@EnableZipkinServer
public class ZipkinApplication {

    public static void main(String[] args) {
        SpringApplication.run(ZipkinApplication.class, args);
    }
}

# （4）配置文件

src/main/resources/applicaiton.properties:

server.port=9010
spring.application.name=zipkin
management.metrics.web.server.auto-time-requests=false

# 六、测试

# 1. 确保启动zipkin server

# 2. 浏览器访问consumer：http://localhost:8090/test, 刷新至少10次

刷新10次是因为，sleuth默认跟踪1/10的请求，因为调用链跟踪是有性能损失的，为了避免对性能产生较大影响，在生成环境中建议把该配置值调低，配置参数：

spring.sleuth.sampler.probability=0.1

注：如果只是为了调试方便，可以把该参数设为1，表示所有请求都进行调用链跟踪

# 3. 浏览器访问provider：http://localhost:8080/test, 刷新至少10次
