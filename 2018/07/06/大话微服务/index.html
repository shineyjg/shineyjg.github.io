大话微服务

2018-07-06
  • 微服务

# 对现有单体系统的改造

  • 不要一下子全部分解，而是先抽取收益最大的部分，采用增量的方式。
  • 而且必须注意到拆分一定会犯错，渐进式重构系统，会让我们有充足时间发现错误并改正。

# 微服务划分

  • 基础服务（日志分析、配置服务器、服务发现、服务授权、分布式追踪、监控）
  • 核心业务服务（避免互相调用）
  • 上层聚合业务服务（允许调用核心服务）

# 微服务大小考量

  • 服务越小，独立性带来的好处越多
  • 但管理大量服务的系统本身也会越复杂

# 架构设计的时候需要确定

  • 是否真的需要强一致性，还是最终一致性。
  • 强一致性： 最好避免事务分布在不同的地方。
  • 如果最终一致性可以接受，则可以拆分成不同服务（不一定非要拆分）

# 犯错无法避免

为了在影响小的地方修正错误。可以在便利贴上模拟服务，每个便利贴写上职责，贴到白板上。 然后虚拟运行尽可能多的实际用例，看：

  • 每个用例用到了哪些服务？
  • 依赖路径是否过长？
  • 循环依赖？
  • 两个服务通讯太密集，是否合并？

# 尽量避免或者禁止不同的微服务强依赖一个独立的源

如某个自己开发的Lib（不管是编译期绑定还是单独jar），除非该Lib非常稳定，而且长时间不会改动，否则升级Lib会导致所有微服务都要升级，这对系统可能造成巨大影响。

总结：

微服务内部贯彻DRY，跨微服务情况可以适当违反DRY

# 异步跨服务传递的信息可能会过期

对信息可用性要求高的场合需要传递引用，如id，接收方收到后需重新根据id同步向特定微服务查询信息。但如果全部按引用来设计可能导致某些查询微服务过热。

# 由于微服务数量多，host也多，物理机也多，必须自动化测试、发布、部署、监控等

# 测试

  • 前期可以多依赖端到端测试
  • 后期可以减少端到端测试，依赖契约测试和线上监控

# 新的部署需要考虑是否采用

  • 蓝绿部署
  • 滚动发布
  • 灰度/金丝雀部署

# 使用关联标识在服务调用之间传递

可以跟踪调用链，解决问题。特别是异步场景，需要在开始时设计。

# 不只规范单个服务，更重要的是要规范整个系统

# 报表

报表可能横跨多个服务，一般而言通过发布订阅消息方式解藕。 对于某些报表系统，可能nosql更合适，比如Neo4j， Cassandra， MongoDB 报表也分仪表盘、告警、普通报告、用户分析等，他们的时效性是不同的，需要使用不同的技术。

# 配置

  • 为了避免配置漂移，禁止任何人登录到服务器手动修改配置。
  • 也就是说配置本身必须和服务一起发布，测试。
  • 即使修改了很小的配置，和修改代码应视为同类改动，也要走持续构建流程。

# 持续构建要求

  • 把代码频繁签入构建分支，建议至少每天签入（为了避免代码提交历史数量众多和混乱，可以在团队中强制git rebase）
  • 有一组测试来验证修改
  • 构建失败时，团队需要以第一优先级修复CI

# 架构师需要专注在大方向上

  • 只在有限的情况下参与到非常具体的细节实现中来。
  • 要保证系统不但能满足当前的需求，还能应对将来的变化。

# 系统长远目标

  • 可以设置一个显示屏，显示基于数据统计的进程。
  • 如用户数，订单数，热门搜索，甚至是营收等关键指标，跟公司战略目标结合，可以明确方向，鼓舞士气。

声明：

上述观点只是一些行业通用做法总结，更多的是一些个人思考，并不是金科玉律，意在抛砖引玉。
